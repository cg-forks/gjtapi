GJTAPI 1.6

Release Notes

Richard Deadman
March 21, 2003

Licencing

GJTAPI is released under an X11/BSD licence, a copy of which can be found on the GJTAPI Sourceforge web site.

Exceptions include the XTAPI bridge code, which is released under LGPL to keep it in line with the GPL licencing of the XTAPI service providers it interfaces with. Other service providers may be released for GJTAPI under other licencing schemes, but the core framework and APIs are under the X11/BSD licence.

Known Limitations

  1. Currently, the following service providers are still in Beta mode:
    • Dialogic - media not yet supported.
    • Voice Modem - modem takes too long to detect when an outgoing call has been answered.
  2. Not all of "callcontrol" is implemented, in particular CallControlTerminal is not supported.

Variances

JTAPI 1.3 Media dictates that MediaService.release() should cause the Connection associated with the MediaService to be dropped. This makes sense for some automatic call routing situations, but not in others where an application may want to tap into an existing call. To provide more flexibility, GJTAPI added a new configuration parameter into version 1.6. If the file "GenericResources.props" contains a line:
mediaReleaseDisconnect = f
then the MediaService will not release the Connection bound to it during "release()".

Installation

GJTAPI consists of three jar files, that should be placed on an application's class path. In addition, the JTAPI jar file should be downloaded from Sun's site. GJTAPI also includes a 1.3 media file that supplies some accidently missing media class files from the JTAPI 1.3 release.

Application Usage

An application must first get a handle on a JTAPI or Jcc Peer and must then ask that peer for a provider. To get the Peer, you use a peer factory and either explicity name the Peer you want or use a default peer. Jtapi looks for default peers by looking for classes with a specific name (i.e. DefaultJtapiPeer). GJTAPI includes these in the core GJTAPI jar file, so if GJTAPI is the first JTAPI implementation listed on your classpath1, your application may use the default method:

JTAPI

Named

JtapiPeer peer = JtapiPeerFactory.getJtapiPeer("net.sourceforge.gjtapi.GenericJtapiPeer");

Default

JtapiPeer peer = JtapiPeerFactory.getJtapiPeer();

JCC

Named

JccPeer peer = JccPeerFactory.getJccPeer("net.sourceforge.gjtapi.jcc.Peer");

Default

JccPeer peer = JccPeerFactory.getJccPeer();
One a handle if had to the Peer, it may be used as a factory to create a provider. This may be done in one of four ways. For simplicity, we show the code for the JTAPI Provider; Jcc is analogous:

Default Provider

Provider prov = peer.getProvider();
This causes the GJTAPI Peer to look in its "GenericResources.props" file for a property "DefaultProvider", which is then looked up using the following algorithm.

Named Provider

Provider prov = peer.getProvider("Emulator");
If there is an entry in "GenericResources.props" called "PROVIDER_Emulator", the value of this property is extracted and used in the following two tests. In the above case, this is resolved to "net.sourceforge.gjtapi.raw.emulator.EmProvider", which is a service provider class. The value "Emulator6", however, points to a property file that indirectly links to the same service provider class, but with a different configuration.
Note that peer.getProviders() will return the set of Provider names listed in the "GenericResources.props" file. GJTAPI allows any service provider to be hooked up, but will not list them unless they are listed in the "GenericResources.props" file.

Props file name

Provider prov = peer.getProvider("Emulator6.props");
If the provider name points to a property file instead of a class, then this file is read to get properties for the provider, including the property "ProviderClass", which is used to find the real service provider class.

Class name

Provider prov = peer.getProvider("net.sourceforge.gjtapi.raw.emulator.EmProvider");
Here we explicitly define the service provider class to hook into GJTAPI.

1 Why would you want more than one JTAPI implementation on you class path? Using the GJTAPI invertor, this allows GJTAPI to make a JTAPI 1.2 implementation look like a JTAPI 1.3 implementation. It also enables two or more JTAPI implementations to be combined together into one virtual telephony space using the GJTAPI multiplexor adapter.
